SQL Schema for MVC Email Flows

Essential database structures for user authentication and messaging features.

A well-designed schema is the foundation of data integrity and application performance.

Core Tables for Users, Tokens & Messages

1. `users` Table
User Accounts & Authentication

Purpose:

Stores all user account information, including authentication credentials and profile details.

`CREATE TABLE` Statement:

CREATE TABLE `users` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `username` VARCHAR(50) UNIQUE NOT NULL,
    `email` VARCHAR(255) UNIQUE NOT NULL,
    `password_hash` VARCHAR(255) NOT NULL,
    `is_verified` BOOLEAN DEFAULT FALSE,
    `two_factor_secret` VARCHAR(255) NULL, -- For 2FA apps
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

Key Columns & Rationale:

`id`: Unique identifier.
`email`: Crucial for logins and password resets. **Must be unique and indexed.**
`password_hash`: Stores the result of `password_hash()` (PHP). **NEVER store plain passwords.**
`is_verified`: For email verification during registration.
`two_factor_secret`: For storing 2FA setup details (e.g., Google Authenticator seed).

2. `password_resets` Table
Password Reset & Email Verification Tokens

Purpose:

Temporarily stores unique tokens for password reset and email verification processes. These tokens are critical for security.

`CREATE TABLE` Statement:

CREATE TABLE `password_resets` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `user_id` INT NOT NULL,
    `token` VARCHAR(255) UNIQUE NOT NULL,
    `type` ENUM('reset', 'verify_email', '2fa_login') NOT NULL,
    `expires_at` TIMESTAMP NOT NULL,
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (`user_id`) REFERENCES `users`(`id`) ON DELETE CASCADE
);

Key Columns & Rationale:

`user_id`: Links the token to a specific user (Foreign Key).
`token`: The unique, cryptographically secure string sent to the user. **Must be unique and indexed for fast lookup.**
`type`: Differentiates between token purposes (e.g., password reset vs. email verification).
`expires_at`: **CRITICAL!** Tokens must expire. Clean up expired tokens regularly.

3. `messages` Table
Internal Mailbox Storage (Sent/Received)

Purpose:

Stores messages for an internal messaging system (e.g., an in-app inbox/sent items), allowing users to write, send, receive, and delete messages.

`CREATE TABLE` Statement:

CREATE TABLE `messages` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `sender_id` INT NOT NULL,
    `recipient_id` INT NOT NULL,
    `subject` VARCHAR(255) NOT NULL,
    `body` TEXT NOT NULL,
    `is_read` BOOLEAN DEFAULT FALSE,
    `is_deleted_by_sender` BOOLEAN DEFAULT FALSE,
    `is_deleted_by_recipient` BOOLEAN DEFAULT FALSE,
    `sent_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (`sender_id`) REFERENCES `users`(`id`) ON DELETE CASCADE,
    FOREIGN KEY (`recipient_id`) REFERENCES `users`(`id`) ON DELETE CASCADE
);

Key Columns & Rationale:

`sender_id`, `recipient_id`: Link to `users` table via Foreign Keys.
`body`: `TEXT` type for potentially long message content.
`is_read`: Status for the recipient's inbox.
`is_deleted_by_sender`, `is_deleted_by_recipient`: **Crucial for soft deletes.** Allows each party to "delete" a message without affecting the other's view, and facilitates recovery.

Developer Focus: Database Best Practices

Beyond these structures, consider these professional practices:

  • **Indexing:** Add indexes to frequently queried columns (e.g., `email` in `users`, `user_id` and `token` in `password_resets`, `recipient_id` in `messages`).
  • **Transactions:** Use database transactions for multi-step operations (e.g., creating a user AND a verification token).
  • **ORM/DBAL:** For PHP, consider using **PDO** directly for simple apps, or an **ORM** (Object-Relational Mapper) like Doctrine or Eloquent for larger projects to abstract SQL.
  • **Regular Cleanup:** Implement a scheduled task (cron job) to remove expired tokens from `password_resets` to maintain security and performance.
Foundational for Scalable & Secure Applications

Integrating with PHP

These tables will be accessed by your **Model layer** in PHP. The Model will contain classes (e.g., `UserModel`, `TokenModel`, `MessageModel`) that encapsulate the logic for interacting with these specific tables using PDO or an ORM.
