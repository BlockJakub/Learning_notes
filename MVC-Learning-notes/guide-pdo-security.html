Secure Model Layer with PDO

Mastering **Prepared Statements** to prevent **SQL Injection**.

Separating SQL logic from user data is the #1 rule in database security.

Two Types of Prepared Statements

1. Named Placeholders (`:name`)
Recommended for Readability

Concept for Beginners:

You assign a meaningful name (prefixed with a colon `:`) inside the SQL query. When you execute the query, you pass an **associative array** where the keys match the placeholder names. Order does not matter, which makes it easier to read.

PHP Code Example:

// 1. Prepare: Use named placeholders
$sql = "INSERT INTO users (email, password_hash) 
        VALUES (:email, :password)";

$stmt = $pdo->prepare($sql);

// 2. Execute: Pass an associative array
$success = $stmt->execute([
    'email' => $userEmail, // Value from user input
    'password' => $hashedPass // Value from hash function
]);

if ($success) {
    $userId = $pdo->lastInsertId();
    // ... continue MVC flow
}

Advanced Benefit:

Ideal for large queries where you might use the same variable multiple times, or where clarity is essential. The array mapping is transparent.

2. Positional Placeholders (`?`)
Simple and Quick for Short Queries

Concept for Beginners:

You simply use a question mark `?` in the SQL where you want the user input to go. When you execute, you pass a simple, **indexed array** where the **order of elements** must exactly match the order of the question marks.

PHP Code Example:

// 1. Prepare: Use positional placeholders
$sql = "UPDATE messages 
        SET is_read = 1 
        WHERE id = ? AND recipient_id = ?";

$stmt = $pdo->prepare($sql);

// 2. Execute: Pass an indexed array (order matters!)
$success = $stmt->execute([
    $messageId, // First '?'
    $currentUserId // Second '?'
]);

if ($stmt->rowCount() > 0) {
    // ... continue MVC flow
}

Advanced Benefit:

Can be marginally faster than named placeholders as the PDO driver does less internal work, but the security guarantee is **exactly the same**. Best used when you have only one or two parameters.

SQL Injection: The Danger We Avoid

The Unsafe Way (Amateur):

// UNSAFE: NEVER DO THIS!
// Attacker Input: 1 OR 1=1
$sql = "SELECT * FROM users WHERE id = " . $_GET['id']; 

// Final SQL sent to DB: 
// SELECT * FROM users WHERE id = 1 OR 1=1
// Result: attacker logs in as the first user (e.g., the Admin)!

The Safe Way (Professional PDO):

// SAFE: The PDO Way
// Attacker Input: 1 OR 1=1
$sql = "SELECT * FROM users WHERE id = ?"; 
$stmt = $pdo->prepare($sql);
$stmt->execute([$_GET['id']]);

// Final SQL sent to DB: 
// The input is treated as a string value, NOT code.
// SELECT * FROM users WHERE id = '1 OR 1=1'
// Result: No user is found, attack fails!

**Key Mechanism:** PDO separates the SQL query template from the actual data. The data is sent to the database later, where it is recognized as a value and automatically quoted/escaped, rendering any malicious SQL code inert.

Advanced Tip: PDO Connection Settings

The PDO Connection ($pdo object) Setup:

When you first create your PDO connection (ideally only once, in your main configuration or database class), you must set crucial attributes for security and reliable error handling.

// DSN (Data Source Name)
$dsn = 'mysql:host=localhost;dbname=your_db;charset=utf8mb4'; 
$options = [
    // Throw exceptions on error for clean error handling (professional standard)
    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION, 
    // Recommended to disable emulation for native MySQL prepared statements
    PDO::ATTR_EMULATE_PREPARES   => false, 
    // Set default fetch mode to associative array
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC, 
];
try {
     $pdo = new PDO($dsn, $user, $pass, $options);
} catch (\PDOException $e) {
     // Log the error internally! Never show detailed errors to the user.
     throw new \PDOException($e->getMessage(), (int)$e->getCode());
}

Where to Use This

Every single database interaction in your **UserModel**, **TokenModel**, and **MessageModel** must use this prepared statement pattern. This includes: `SELECT`, `INSERT`, `UPDATE`, and `DELETE` queries that involve any user-provided data.
