MVC Request-Response Cycle (PHP Deep Dive)

A professional, 7-stage architectural breakdown for building robust web applications.

Separation of Concerns is the foundation of scalable development.

The 7 Phases of the Request Lifecycle

1. USER (Browser)
Client-Side Interaction

Amateur/High-Level

You ask for a page by typing its web address (URL) and hitting Enter.

Intermediate/Code-Focused

The browser initiates an **HTTP Request** (GET/POST/PUT/DELETE) to the server, targeting a specific **Uniform Resource Identifier (URI)**.

Professional/Jargon

The client-side user-agent transmits an **HTTP verb** and **URI** to the origin server's **IP address**, initiating the **TCP handshake** required for the application lifecycle.

2. Web Server / Router
Traffic & Mapping

Amateur/High-Level

A bouncer or doorman checks your destination and sends you to the correct internal department.

Intermediate/Code-Focused

The **Web Server** (Apache/Nginx) uses a rule (via `.htaccess` or similar config) to direct **all** traffic to a single **Front Controller** file (usually `index.php`). The **Router** then analyzes the URI to identify the target **Controller class** and **action method**.

Professional/Jargon

The request is bootstrapped through a single entry point. A **Middleware** layer may execute pre-processing logic (e.g., rate limiting, session initiation) before the **Routing Service** dispatches the request to the designated **Controller FQCN** and method based on a configured **Route Map**.

3. CONTROLLER ("C" in MVC)
Coordination Layer

Amateur/High-Level

The Controller is the manager. It takes the order, talks to the experts (Model), and gets ready to present the final result (View).

Intermediate/Code-Focused

The Controller receives the **Request object**. It executes primary application logic, calls the necessary **Model methods** (e.g., dependency injection of a repository), and marshals the resulting data for the View.

Professional/Jargon

The Controller handles the **Request/Response lifecycle**. It delegates data persistence tasks to the **Service Layer** or **Repositories** (Model) and determines the appropriate **View/Template** to render the response. It MUST remain lean, focusing solely on request delegation.

4. MODEL ("M" in MVC)
Business Logic / Domain Layer

Amateur/High-Level

The expert. It knows all the rules about the data and how to talk to the database. It finds the data and brings it back.

Intermediate/Code-Focused

The Model encapsulates the **Business Logic** and **Data Access Logic (DAL)**. It uses **ORM (Object-Relational Mapping)** to retrieve or save entities, ensuring data integrity via built-in validation rules.

Professional/Jargon

The Model comprises the **Domain Layer**. It includes **Entities**, **Value Objects**, and **Repositories**. Repositories abstract the DAL by mapping domain object interactions to persistence operations, shielding the Controller from **SQL queries** or schema details.

5. DATABASE (DB)
Data Persistence Layer

Amateur/High-Level

The giant filing cabinet. It stores everything and gives back the files when the Model asks for them.

Intermediate/Code-Focused

Receives **SQL queries** (SELECT, INSERT, UPDATE, DELETE) generated by the **Model/ORM**. It executes the query and returns a **Result Set** (raw data) to the Model.

Professional/Jargon

This layer manages **data persistence** and ensures **ACID properties**. The Model leverages the **PDO** or a native driver to execute queries, often managing transactions to maintain data consistency during complex operations.

6. VIEW ("V" in MVC)
Presentation Layer / Templating

Amateur/High-Level

The View is the final presentation layout. It takes the data and displays it nicely using HTML and CSS.

Intermediate/Code-Focused

The View receives the **marshaled data** from the Controller. It uses a **Templating Engine** (e.g., Twig, Blade) to embed data into the static HTML structure, focusing only on **display logic** (loops, conditionals).

Professional/Jargon

The View renders the **Representation** of the resource. It must strictly adhere to the **Single Responsibility Principle**, containing no Business Logic. Output is generated as a buffer, ready for final assembly into the **HTTP Response body**.

7. HTML RESPONSE (Final Delivery)

The browser receives the final HTML and CSS.

The generated **HTML stream** is packaged into an **HTTP Response** by the PHP application and sent back through the Web Server to the User's browser. The browser then parses and renders the final web page.

Final Output: User Interface

Developer Insight: Separation of Concerns

The goal of this entire architecture is to prevent **"spaghetti code."** By enforcing that the **Controller** manages the flow, the **Model** handles data rules, and the **View** handles presentation, you achieve a clean, testable, and maintainable application structure.
